# Production Animation Pipeline Design

## Goal

Build a production path for a pose-controlled pixel-art sword attack that keeps
the character temporally coherent, treats the weapon as invariant rigid
artwork, and exports a directly loadable Godot 4.x animation bundle.

The first production version supports the built-in `sword_attack` action,
`8`, `12`, or `16` output frames, and sprite sizes `64`, `96`, `128`, and
`256`. It preserves the existing workflows and `/v1/game-assets` behavior.

## Decision

Use a hybrid pipeline:

1. Plan a dense, phase-aware human and weapon trajectory from authored action
   keyframes.
2. Generate the complete weapon-free character sequence as one temporal SDXL
   batch with AnimateDiff, OpenPose ControlNet, and IP-Adapter.
3. Stabilize and palette-normalize the RGBA character frames.
4. Apply one user-supplied transparent weapon image to every frame with a
   deterministic similarity transform.
5. Export individual frames, a sprite sheet, a preview, an engine-neutral
   metadata file, and a Godot 4.x `SpriteFrames` resource.

The weapon is never generated by the diffusion model. This is the required
boundary that prevents blade shape, length, grip, and palette drift.

## Alternatives Considered

### Sequential SDXL img2img

Feeding each accepted frame into the next frame needs fewer new components, but
errors accumulate through the sequence and rapid motion tends to produce
ghosting. It also serializes generation and makes a bad early frame affect every
later frame.

### Pure 2D skeletal rendering

Rigging separated character parts gives the strongest geometric guarantees,
but requires manual part extraction for every character and handles robes,
hair, and other non-rigid shapes poorly. It does not meet the goal of automated
production from a character reference.

- `Kosinkadink/ComfyUI-AnimateDiff-Evolved` pinned to commit
  `d8d163cd90b1111f6227495e3467633676fbb346` (Apache-2.0).
- `guoyww/animatediff-motion-adapter-sdxl-beta` revision
  `26c864717b4d4b002bb48ae6c9d6bb431548c6cb`, source file
  `diffusion_pytorch_model.fp16.safetensors`, installed as
  `models/animatediff_models/mm_sdxl_v10_beta.safetensors`.
  The file is `474328896` bytes with SHA-256
  `24c3c5f48006ce2ce7b06188622865c620b2d33db23b1af671cc1f21716b5826`.

The existing resumable, hash-verifying model installer and pinned node archive
installer own deployment. They use the configured Hugging Face mirror first
and upstream Hugging Face as an explicit fallback. No core ComfyUI module gains
network behavior.

Discovery must require these current node types from the pinned node revision:

- `ADE_LoadAnimateDiffModel`
- `ADE_ApplyAnimateDiffModelSimple`
- `ADE_UseEvolvedSampling`
- `ADE_StandardUniformContextOptions`

Existing SDXL Base, Pixel Art XL, OpenPoseXL2, IP-Adapter Plus SDXL, ViT-H
clip-vision, and BiRefNet artifacts remain unchanged.

## API Contract

Add `POST /v1/animations`. Existing routes retain their current request and
response contracts. Animation jobs use the existing `GET /v1/jobs/{job_id}`
and `/assets/{job_id}/{path}` routes.

The production request is JSON:

```json
{
  "asset_name": "cultivator_attack",
  "character_image": "characters/cultivator.png",
  "character_prompt": "side-view cultivation swordsman in white and cyan robes",
  "weapon": "weapons/sword.json",
  "action": "sword_attack",
  "frame_count": 12,
  "sprite_size": 128,
  "seed": 42,
  "godot_resource_prefix": "res://game_assets/cultivator_attack"
}
```

`asset_name`, `character_image`, `character_prompt`, `weapon`, and `action` are
required. `action` accepts only `sword_attack` in this version. `frame_count`
defaults to `12`, `sprite_size` defaults to `128`, and `seed` defaults to a
generated unsigned 64-bit integer. `godot_resource_prefix` defaults to
`res://game_assets/<asset_name>`.

`asset_name` accepts ASCII letters, digits, `_`, and `-`. Input paths are
relative to the configured ComfyUI `input/` directory, may not contain `..`,
and must resolve beneath that directory. The Godot prefix must begin with
`res://`, may not contain `..`, and uses forward slashes.

The creation response uses the current asynchronous job shape:

```json
{
  "job_id": "<uuid>",
  "status": "queued"
}
```

The completed job response adds URLs for `frames`, `spritesheet`,
`sprite_frames`, `metadata`, and `preview` without changing existing job
responses.

## Weapon Descriptor

The user supplies one RGBA PNG and one JSON descriptor. The image path is
resolved relative to the descriptor and remains subject to the `input/`
boundary.

```json
{
  "schema_version": 1,
  "image": "sword.png",
  "grip": [0.16, 0.52],
  "tip": [0.94, 0.48],
  "default_layer": "behind_character"
}
```

`grip` and `tip` are normalized source-image coordinates. Every coordinate
must be in `[0, 1]`; the two points must not be equal. `default_layer` is
`behind_character` or `in_front_of_character`. The descriptor carries no
arbitrary file, node, expression, or script content.

## Motion Planning

Represent the sword attack as authored phase keyframes:

- anticipation
- draw back
- wind up
- acceleration
- contact
- follow through
- overshoot
- recovery

Each keyframe owns OpenPose joints, a target weapon grip, a target weapon tip,
the weapon layer, normalized time, and optional events. `contact` owns the
`hit` event. The attack is non-looping.

For an output count of `N`, construct `3N` motion samples. Human joints use
piecewise cubic Hermite interpolation with endpoints constrained to the
authored phase values. Foot-contact phases keep the planted ankle fixed. The
pelvis follows an explicit root curve rather than being inferred from limb
positions. Weapon grip position uses the same phase timing, while weapon angle
uses shortest-path angular interpolation so it follows one continuous arc.

Select `N` final motion samples with phase-aware timing. Anticipation and
recovery retain more display time, acceleration uses the shortest durations,
and contact has one explicit hold. The selected samples provide the exact
OpenPose batch, target weapon transforms, root coordinates, frame durations,
and event positions used by all downstream stages.

## Temporal ComfyUI Graph

Build one API graph for the complete selected frame sequence:

1. Load SDXL Base and apply Pixel Art XL.
2. Apply IP-Adapter Plus SDXL to the approved character reference for the full
   sequence.
3. Load the selected pose images as one image batch and apply OpenPoseXL2 to the
   matching conditioning batch.
4. Load `mm_sdxl_v10_beta.safetensors`, apply it through AnimateDiff-Evolved,
   and use a uniform context length of `8` with overlap `2` for `12`- and
   `16`-frame requests. Eight-frame requests use one context.
5. Sample a matching latent batch at `512x512` with one request seed.
6. Decode and remove backgrounds with the existing BiRefNet path.

The positive prompt describes the character, fixed side camera, action phase,
and pixel-art style. The negative prompt explicitly rejects swords, weapons,
scabbards, duplicate limbs, camera changes, and cropped bodies. The model is
responsible for the character only; invisible weapon targets are represented
by wrist and elbow placement in the OpenPose batch.

No independent-frame fallback is allowed. Missing temporal nodes, an invalid
motion module, or an out-of-memory failure terminates the production job with
its stage recorded.

## Character Stabilization

Postprocessing operates on the returned RGBA batch before weapon composition:

- Validate identical dimensions, a non-empty foreground, and transparent
  background in every frame.
- Align each grounded frame's foreground bottom-center to the planned root
  target, using integer pixel translations only.
- Use one fixed canvas and reject a sequence if any aligned foreground would be
  clipped.
- Build one palette from the complete sequence, reserve transparent index zero,
  and quantize every frame against that shared palette.
- Resize to the requested sprite size with nearest-neighbor sampling.

The alignment transform is recorded per frame. This stage does not synthesize
pixels, interpolate output frames, or hide an invalid generation result.

## Weapon Composition

For each selected frame, map the descriptor's source `grip -> tip` segment to
the motion plan's target `grip -> tip` segment with one similarity transform:

- uniform scale from segment-length ratio;
- shortest-angle rotation from segment direction;
- translation that maps the source grip exactly to the target grip.

Apply the transform with nearest-neighbor sampling on a transparent canvas.
Composite the transformed weapon before or after the character according to
the phase layer. The same source pixels and shared output palette are used in
every frame. Record the scale, rotation, translation, transformed grip, and
transformed tip in metadata.

The compositor rejects a transformed weapon that is clipped or whose grip or
tip differs from the requested target by more than one output pixel.

## Output Bundle

Publish only this completed directory:

```text
output/game_assets/<job_id>/production_action/
  frames/000.png
  frames/001.png
  ...
  spritesheet.png
  sprite_frames.tres
  animation.json
  preview.gif
```

`animation.json` uses `schema_version: 1` and contains:

- asset name, action, non-looping flag, canvas size, and frame count;
- sprite-sheet grid and frame regions;
- per-frame duration in seconds;
- phase and events, including `hit` on contact;
- planned root and applied alignment translation;
- target and transformed weapon endpoints and similarity transform;
- relative artifact filenames.

`sprite_frames.tres` is a Godot 4.x text `SpriteFrames` resource. It references
`<godot_resource_prefix>/spritesheet.png`, defines atlas regions for every
frame, stores relative duration multipliers, and sets `loop = false`. The base
animation speed is `12.0` FPS. Copying the published bundle to the configured
Godot resource prefix makes the `.tres` loadable without rewriting paths.

The preview GIF follows the same per-frame durations as the Godot resource.

## Job Stages And Atomic Publication

Production jobs expose these ordered stages:

1. `validating_inputs`
2. `motion_planning`
3. `temporal_generation`
4. `character_stabilization`
5. `weapon_composite`
6. `godot_export`
7. `validating_outputs`
8. `completed`

All work occurs under a job-specific temporary directory. The final
`production_action` directory is published with one rename only after every
stage and acceptance check succeeds. Failures retain the job record and stage
with a short actionable message, delete the temporary output, and never expose
partial bundle URLs.

## Validation And Testing

Unit tests cover:

- request defaults, invalid enums, safe input paths, and safe Godot prefixes;
- weapon descriptor parsing and confinement;
- `3N` motion sample counts for `N = 8, 12, 16`;
- exact phase endpoints, planted-foot constraints, continuous weapon angles,
  contact event placement, and positive frame durations;
- source-to-target weapon endpoint mapping within one pixel;
- shared palette use, integer alignment, clipping rejection, and stable output
  dimensions;
- sprite-sheet regions, animation metadata, and Godot resource text.

Workflow tests assert current AnimateDiff-Evolved node names and graph inputs
against a representative `/object_info` fixture. Existing workflow and API
tests remain unchanged and passing.

Deployment validation runs in this order:

1. Install and verify the pinned node and motion module.
2. Restart ComfyUI and validate all required node and model selections through
   `/object_info`.
3. Run a two-frame, 64-pixel temporal preflight to detect dependency or VRAM
   failures before a full job.
4. Generate fresh `8`, `12`, and `16` frame sword attacks at 128 pixels.
5. Load `sprite_frames.tres` with a configured Godot 4.x binary using headless
   mode; the validator rejects a missing binary or a non-4.x engine.

Automated acceptance requires:

- requested frame counts and dimensions throughout every artifact;
- non-empty RGBA foreground and transparent background in every frame;
- no aligned foreground or weapon clipping;
- root alignment error no greater than one output pixel;
- weapon grip and tip error no greater than one output pixel;
- one shared sequence palette and the same source weapon artifact in all frames;
- matching GIF, metadata, sprite-sheet, and Godot timing;
- successful Godot 4.x headless load;
- the complete Python regression suite passing.

The final visual review checks identity, clothing, hair, limb topology, hand
occlusion, planted feet, anticipation, contact, follow-through, recovery, and
weapon arc. Automated geometry checks cannot replace this review.

## Compatibility And Security

- Existing API calls, workflow filenames, model paths, and deployed ComfyUI
  behavior remain valid.
- The new endpoint remains loopback-only with the current service.
- Input and output paths are confined to their configured roots.
- JSON descriptors never select arbitrary nodes, commands, or Python imports.
- Downloads are explicit deployment actions, pinned to the requested artifacts,
  and include no telemetry or background update check.
- Core ComfyUI stays on the official repository and receives no source changes.

## Non-Goals

This version does not add arbitrary action authoring, other weapon families,
automatic weapon extraction, FaceID, a ComfyUI frontend editor, Unity or Godot
3.x export, runtime skeletal animation, frame interpolation after pixelization,
or a silent non-temporal fallback.
